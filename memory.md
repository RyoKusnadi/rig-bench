# Complete Implementation Plan: Unified Memory System for `rig-bench`

This is the comprehensive, phased implementation plan that merges all the memory concepts we've designed. It is structured so you can hand this document to an AI agent and have it execute the implementation step-by-step. All tasks, including initial documentation, are designed to be fully AI-driven.

---

## Phase 0: Foundation & Bootstrap

**Objective:** Set up the basic memory structure and seed it with initial data so the system has immediate value.

**Dependencies:** This phase must be completed before any other phase.

### Deliverable 0.1: Create Memory Directory Structure

Create the memory vault directory structure in the repository root.

**Directory structure to create:**
- `memory/` - Root directory for all memory artifacts
- `memory/RULES.md` - Semantic memory file for coding standards (empty initially)
- `memory/ARCHITECTURE.md` - Semantic memory file for high-level system design (empty initially)
- `memory/structure.json` - Structural index file (initialize as empty JSON array)
- `memory/PENDING_UPDATES.md` - File for drift detection alerts (empty initially)
- `memory/archive/` - Directory for episodic memory
- `memory/archive/git/` - Subdirectory for Git commit index
- `memory/archive/git/index.json` - Git history index (initialize as empty JSON array)
- `memory/archive/summaries/` - Subdirectory for on-demand file summaries

**Verification:**
- Verify all directories exist
- Verify all JSON files contain valid empty arrays
- Verify all Markdown files exist (can be empty)

### Deliverable 0.2: AI-Driven Semantic Memory Generation

**Objective:** Use an AI agent to analyze the existing codebase and automatically generate the initial `RULES.md` and `ARCHITECTURE.md` files, eliminating the need for manual human curation.

**Implementation requirements:**
1. **Architect Agent Spawning:** Create a bootstrap workflow that spawns a dedicated "Architect Agent" (using a high-reasoning model). This agent is given read-only access to the codebase.
2. **Context Provisioning:** Provide the Architect Agent with a high-level view of the project:
   - The full directory tree (excluding dependencies and build artifacts).
   - The contents of package manifests (e.g., `package.json`, `requirements.txt`) to understand dependencies.
   - The contents of configuration files (e.g., `tsconfig.json`, `.eslintrc`, CI/CD pipelines) to understand constraints.
   - The contents of key entry-point files (e.g., `main.ts`, `index.js`, `app.py`).
3. **Generating `ARCHITECTURE.md`:** Instruct the Architect Agent to analyze the provided context and generate a comprehensive architectural document. The agent must:
   - Identify and define all major modules, services, or domains.
   - Map the primary data flow (how requests enter, how they are processed, and where data is stored).
   - List key files and their specific responsibilities.
   - Document external integrations and third-party services.
4. **Generating `RULES.md`:** Instruct the Architect Agent to infer and document the project's coding standards by analyzing the actual codebase. The agent must:
   - Identify established patterns for error handling, logging, and asynchronous operations.
   - Document naming conventions and file organization rules observed in the code.
   - Extract testing requirements and conventions (e.g., test file naming, mocking strategies).
   - List strict constraints derived from configuration files (e.g., strict typing rules, linting constraints).
5. **AI-Driven Validation (Self-Correction):** To mitigate hallucinations, spawn a second "Reviewer Agent" (or reuse the Architect Agent in a fresh context). The Reviewer Agent is given the generated `RULES.md` and `ARCHITECTURE.md`, along with a random sample of 5-10 source files. It must verify that the documented rules and architecture actually match the code. If it finds discrepancies, it corrects the documents directly.
6. **Finalization:** The validated files are saved to `memory/RULES.md` and `memory/ARCHITECTURE.md`.

**Verification:**
- Verify both files exist and contain substantive content generated by the AI.
- Verify `RULES.md` accurately reflects the coding patterns found in the codebase.
- Verify `ARCHITECTURE.md` correctly maps the major modules and data flow.

### Deliverable 0.3: Git History Bootstrap Script

Create a script that indexes the last 50 commits into a searchable format.

**Script purpose:** Extract recent Git history and store it in a structured, searchable index.

**Implementation requirements:**
1. Use Git log commands to retrieve the last 50 commits.
2. For each commit, extract three pieces of information:
   - Commit SHA (the unique identifier)
   - Commit message (the description of what was done)
   - List of files changed (all files modified in that commit)
3. Store this data as a JSON array in `memory/archive/git/index.json`.
4. Format each entry with the three fields: sha, message, and files (comma-separated list).

**Why 50 commits?** This provides enough historical context for the agent to find relevant patterns without overwhelming the search results. It can be increased later if needed.

**Verification:**
- Run the script
- Verify `memory/archive/git/index.json` contains valid JSON
- Verify it contains up to 50 commit entries
- Verify each entry has all three required fields
- Verify the data is accurate by spot-checking a few commits against actual Git history

## Phase 1: Structural Memory & Indexing

**Objective:** Build the "map" of the codebase so the agent knows where to look, eliminating blind searches.

**Dependencies:** Phase 0 must be complete. Phase 2 depends on this phase.

### Deliverable 1.1: Structural Index Script

Create a script that generates a structural map of the entire codebase.

**Script purpose:** Scan all source files and extract their structure (exports, imports, file paths) into a searchable index.

**Implementation requirements:**
1. Scan all source files in the repository, excluding:
   - `node_modules/` (or equivalent dependency directories)
   - `dist/`, `build/` (compiled output directories)
   - `.git/` (Git metadata)
   - `memory/` (the memory system itself)
2. For each file, extract four pieces of information:
   - File path (relative to repository root)
   - File type or extension (e.g., "typescript", "javascript", "python")
   - Exported functions and classes (what this file provides to other files)
   - Import statements (what this file depends on from other files)
3. Output the results to `memory/structure.json` as a JSON array.
4. Format each entry with the four fields: path, type, exports (array), imports (array).

**Implementation approach:**
- Use file system commands to list all relevant files.
- Use pattern matching or regular expressions to extract exports and imports.
- For TypeScript/JavaScript files: look for export statements (export function, export class, export const).
- For import statements: look for import...from patterns.
- Parse the extracted data and format it as JSON.

**Why this matters:** This gives the agent a "GPS" for the codebase. Instead of blindly searching with ripgrep and getting thousands of results, the agent can query the structural map and immediately know which files are relevant.

**Verification:**
- Run the script
- Verify `memory/structure.json` contains valid JSON
- Verify it includes files from the codebase
- Verify exports and imports are extracted correctly by spot-checking a few files
- Verify the script completes in a reasonable time (should be under 10 seconds for medium codebases)

### Deliverable 1.2: Pre-Flight Sync Hook

Modify the Operator workflow to ensure the structural index is always fresh before task execution.

**Hook purpose:** Automatically regenerate the structural index before every task to guarantee it's 100% accurate.

**Implementation requirements:**
1. Add a hook that runs immediately before spawning the Operator agent.
2. The hook executes the structural index script.
3. This ensures `memory/structure.json` is synced with the current HEAD of the main branch.
4. Add error handling: if the script fails, log the error but continue (don't block the task).

**Why pre-flight sync?** The codebase changes constantly. Files are added, moved, deleted, and modified. By regenerating the map before every task, we guarantee the agent always has an accurate view of the codebase structure. The script is fast and costs zero tokens, so there's no downside to running it every time.

**Verification:**
- Create a test spec
- Run the workflow
- Verify `memory/structure.json` is updated before the Operator spawns
- Verify the task completes successfully
- Verify that if the script fails, the task still proceeds (with a logged error)

### Deliverable 1.3: Structural Search Tool

Create a tool that allows the agent to query the structural map.

**Tool purpose:** Enable the agent to find files, functions, and dependencies by searching the structural index.

**Implementation requirements:**
1. Accept a query string as input.
2. Search `memory/structure.json` for matches in:
   - File paths
   - Exported function/class names
   - Import paths
3. Return the top 5 matches as formatted text.
4. Format the output to be easily readable by the LLM, showing:
   - File path
   - What the file exports
   - What the file imports

**How the agent uses this:** When the agent receives a spec like "Add invoice cancellation endpoint", it calls this tool with the query "invoice". The tool returns the relevant files, so the agent knows exactly where to look without blind searching.

**Register as agent tool:**
- Add to the Operator agent's tool definitions
- Tool name: `search_structure`
- Description: "Search the codebase structure to find files, functions, and dependencies"
- Input: query string
- Output: formatted search results

**Verification:**
- Run the tool with various queries
- Verify it returns relevant results
- Test with the agent: have it call `search_structure("invoice")` and verify it gets useful results
- Verify the output format is clear and parseable by the LLM

### Deliverable 2.2: On-Demand File Summary Tool with Hash-Based Invalidation

Create a tool that provides cached file summaries with automatic invalidation when files change.

**Tool purpose:** Allow the agent to quickly understand complex files without re-reading them every time, while ensuring the summaries are always fresh.

**Implementation requirements:**
1. Accept a file path as input.
2. Calculate the current Git blob hash of the file (this is a unique fingerprint of the file's content).
3. Check if a cached summary exists in `memory/archive/summaries/`.
4. Check if the cached hash matches the current hash.
5. **If hash matches:** Return the cached summary (saves massive tokens).
6. **If hash doesn't match or no cache exists:**
   - Read the raw file
   - Generate a summary using an LLM call (this is the only LLM call in the entire memory system)
   - Save the summary to `memory/archive/summaries/` with a filename based on the file path
   - Save the hash to a corresponding `.hash` file
   - Return the new summary

**Summary generation (LLM call):**
- When a new summary needs to be generated, invoke the LLM with a prompt that asks for:
  - What the file does (purpose)
  - Key functions/classes and what they do
  - Important dependencies
  - Any notable patterns or gotchas
- Keep the summary concise (around 200 words)
- Use a fast, cheap model (like Haiku) for this

**Cache storage:**
- Summary file: stored in `memory/archive/summaries/` with a filename derived from the file path
- Hash file: stored alongside the summary with the same base name but `.hash` extension
- Use a hash of the file path as the filename to avoid issues with special characters or long paths

**Why hash-based invalidation?** This is the self-healing mechanism that ensures the agent never reads stale information. If a file changes, the hash changes, and the cache is automatically invalidated. This requires zero LLM tokens to verify freshness.

**Register as agent tool:**
- Add to the Operator agent's tool definitions
- Tool name: `read_file_summary`
- Description: "Read a file summary (cached) or the raw file if not cached. Automatically generates and caches summaries for complex files."
- Input: file path
- Output: file summary or raw file content

**Verification:**
- Run the tool on a file that hasn't been cached
- Verify it generates a summary and saves it
- Verify the hash is saved
- Run the tool again on the same file
- Verify it returns the cached summary
- Modify the file
- Run the tool again
- Verify it detects the hash mismatch and regenerates the summary

### Deliverable 2.3: Spec Archiving Script

Create a script that archives completed specs into the episodic memory.

**Script purpose:** Automatically preserve completed specs and their associated Git commits for future reference.

**Implementation requirements:**
1. Accept a spec ID as input.
2. Copy the spec file from `specs/finished/` to `memory/archive/<spec-id>/spec.md`.
3. Extract the commit SHA from the Git log (the commit that closed the spec).
4. Save the commit SHA to `memory/archive/<spec-id>/commit.sha`.
5. Append metadata to an index file `memory/archive/index.json` containing:
   - Spec ID
   - Title (extracted from the spec's frontmatter)
   - Tags (extracted from the spec's frontmatter)
   - Commit SHA
   - Timestamp of when it was archived

**Integration point:**
- Call this script from the Shipper workflow, after the PR is created but before the worktree is deleted
- Alternatively, call it from a post-task hook that runs after the Inspector passes the task

**Why this matters:** Every completed spec becomes part of the episodic memory. Future agents can search this archive to see how similar features were implemented.

**Verification:**
- Complete a test spec
- Verify the spec is copied to `memory/archive/`
- Verify the commit SHA is recorded
- Verify the index is updated with the correct metadata

## Phase 3: Inspector Enhancements & Drift Detection

**Objective:** Make the Inspector agent an automated auditor that detects when semantic memory is outdated.

**Dependencies:** Phase 0, 1, and 2 must be complete. This phase is independent of Phase 4.

### Deliverable 3.1: Diff-Only Inspection

Modify the Inspector workflow to prioritize reading diffs over full files.

**Purpose:** Reduce token consumption during verification by focusing on what changed, not what didn't.

**Implementation requirements:**
1. Add instruction to the Inspector agent's system prompt:
   - When verifying code changes, the Inspector MUST use the `read_worktree_diff` tool first
   - Only read full files if the diff is ambiguous or references undefined symbols
   - Never read entire files unless absolutely necessary

2. Create a tool that reads the worktree diff:
   - Run `git diff main...HEAD` in the current worktree
   - Return the diff output
   - Truncate if the diff is larger than 10,000 lines to avoid context overflow
   - Format as standard Git diff output

3. Register as agent tool:
   - Tool name: `read_worktree_diff`
   - Description: "Read the Git diff of changes in the current worktree"
   - Input: none (uses current worktree)
   - Output: Git diff

**Why this matters:** The Inspector previously read entire files to verify changes, which consumed massive amounts of tokens. By focusing on the diff, we reduce verification tokens by 70-80%.

**Verification:**
- Create a test spec that modifies 3 files
- Run the Inspector
- Verify it calls `read_worktree_diff` first
- Verify it only reads full files if necessary (check logs)
- Verify the task completes successfully
- Verify token consumption is significantly lower than before

### Deliverable 3.2: Drift Detection Prompt

Modify the Inspector agent's system prompt to detect when semantic memory is outdated.

**Purpose:** Automatically identify when the codebase has evolved in ways that contradict the documented architecture or rules.

**Implementation requirements:**
1. Add instruction to the Inspector's system prompt:
   - While verifying the diff, check if the implementation fundamentally contradicts or changes the high-level rules in `ARCHITECTURE.md` or `RULES.md`
   - Examples of architectural shifts:
     - Changing a database schema
     - Introducing a new external API
     - Changing authentication flow
     - Modifying core data structures
   - If a major architectural shift occurred that is NOT documented in the memory files, the Inspector must output a `MEMORY_DRIFT_WARNING` in its final report
   - The warning should include a description of what changed and what needs to be updated

**Why this matters:** Semantic memory is AI-generated and can drift as the codebase evolves. The Inspector acts as an automated auditor that catches when the docs fall out of sync with reality, alerting the system to update them.

**Verification:**
- Create a test spec that fundamentally changes the architecture
- Run the Inspector
- Verify it outputs `MEMORY_DRIFT_WARNING` in the report
- Verify the warning accurately describes the drift

### Deliverable 3.3: Drift Alert Handler

Modify the harness to capture drift warnings and trigger an AI update.

**Purpose:** Ensure drift warnings are recorded and automatically handled.

**Implementation requirements:**
1. After the Inspector completes, parse its output for `MEMORY_DRIFT_WARNING`.
2. If found:
   - Extract the warning message
   - Append to `memory/PENDING_UPDATES.md` with a timestamp and the spec ID
   - Format the entry clearly with the warning description
   - Output to console with a warning symbol and a message pointing to the file
3. **Automated Resolution:** Spawn a "Maintenance Agent" (using a fast model) to read the `PENDING_UPDATES.md` file, analyze the recent Git diff that caused the drift, and automatically rewrite the outdated sections in `RULES.md` or `ARCHITECTURE.md`. Once updated, clear the entry from `PENDING_UPDATES.md`.

**Why this approach?** Since we prefer everything using AI, we don't just alert a human. We use an AI agent to automatically fix the semantic memory drift based on the Inspector's findings.

**Verification:**
- Run a task that triggers drift detection
- Verify `memory/PENDING_UPDATES.md` is updated
- Verify the Maintenance Agent spawns and updates the semantic files
- Verify the entry is cleared from `PENDING_UPDATES.md` after resolution

## Phase 4: Context Management & Optimization

**Objective:** Handle long-running tasks and optimize token usage for different complexity levels.

**Dependencies:** Phase 0, 1, 2, and 3 must be complete. This phase is independent.

### Deliverable 4.1: Checkpoint Pattern for Long Tasks

Modify the Operator workflow to handle tasks that exceed context window limits.

**Purpose:** Allow the agent to pause and resume long tasks without losing progress.

**Implementation requirements:**
1. Add instruction to the Operator agent's system prompt:
   - If the agent feels its context is getting too large, it should write a checkpoint
   - The checkpoint process:
     - Create a file called `PROGRESS.md` in the worktree root
     - Write a summary of what has been accomplished so far
     - Write what needs to be done next
     - Commit the file
     - Stop execution

2. Modify the harness to detect checkpoints:
   - After the Operator stops, check if `PROGRESS.md` exists in the worktree
   - If it exists:
     - Spawn a new Operator agent instance
     - Inject the contents of `PROGRESS.md` into the system prompt
     - The new agent continues from the checkpoint
   - If it doesn't exist:
     - The task is complete, proceed to Inspector

**Why checkpoint instead of compact_context?** Claude Code's API doesn't expose conversation history manipulation. The checkpoint pattern is less elegant but actually works with the current constraints.

**Verification:**
- Create a test spec that touches 20+ files
- Run the workflow
- Verify the Operator writes `PROGRESS.md` when context gets large
- Verify a new agent instance is spawned
- Verify the new agent reads the checkpoint and continues
- Verify the task completes successfully

### Deliverable 4.2: Complexity-Based Memory Loading

Modify the Operator workflow to load different amounts of context based on task complexity.

**Purpose:** Reduce token consumption for simple tasks by not loading unnecessary memory.

**Implementation requirements:**
1. Before spawning the Operator, evaluate the spec's complexity:
   - Check the `complexity` field in the spec's YAML frontmatter (if present)
   - Or estimate complexity based on the `files_to_modify` array length
2. **If complexity is "low" or touches fewer than 3 files:**
   - Inject minimal context: only `CLAUDE.md`
   - Do NOT inject `RULES.md` or `ARCHITECTURE.md`
3. **If complexity is "high" or touches 3 or more files:**
   - Inject full context: `CLAUDE.md` + `RULES.md` + `ARCHITECTURE.md`

**Implementation approach:**
- Parse the spec's YAML frontmatter
- Count the number of files in `files_to_modify` (if present)
- Use conditional logic to determine which files to inject

**Why this matters:** Simple tasks don't need to load the entire architectural context. By loading minimal context for simple tasks, we save tokens and make the agent faster.

**Verification:**
- Create a simple spec
- Verify the Operator receives minimal context
- Create a complex spec
- Verify the Operator receives full context
- Verify both tasks complete successfully

### Deliverable 4.3: Time-Decay for Git History

Modify the Git history search tool to prevent use of outdated patterns from old commits.

**Purpose:** Ensure the agent doesn't use patterns from commits that are no longer relevant due to framework updates or major refactoring.

**Implementation requirements:**
1. When searching commits, filter out commits older than 6 months:
   - Use Git log with a date filter when building the index
   - Or filter in the search script based on commit date
2. Check for major version bumps in dependency files:
   - Parse package manifests
   - If a major version bump is detected, flag all commits before that date as `[LEGACY]`
3. When returning search results, append `[LEGACY]` tag to old commits.
4. Add instruction to Operator agent's system prompt:
   - Never use patterns from commits tagged as `[LEGACY]`
   - These commits use outdated APIs or frameworks

**Why this matters:** A commit from 2 years ago might have used an API that no longer exists. By tagging old commits as legacy, we prevent the agent from confidently implementing deprecated patterns.

**Verification:**
- Run a search that returns both recent and old commits
- Verify old commits are tagged as `[LEGACY]`
- Verify the agent avoids using legacy patterns

## Troubleshooting Guide

### Issue: Structural index is empty
**Cause:** Script failed to scan files
**Fix:** Check file permissions, verify file listing commands work, check for syntax errors in the script

### Issue: File summaries are stale
**Cause:** Hash comparison failed
**Fix:** Verify Git hash commands work, check that hash files are being saved correctly

### Issue: Inspector doesn't detect drift
**Cause:** Prompt instruction not clear enough
**Fix:** Make the drift detection instruction more explicit, provide examples

### Issue: Checkpoint pattern doesn't trigger
**Cause:** Operator doesn't recognize when context is large
**Fix:** Add explicit instruction to monitor tool call count or context size

### Issue: Token consumption is still high
**Cause:** Memory tools returning too much data
**Fix:** Limit search results to top 3-5 matches, truncate long outputs
